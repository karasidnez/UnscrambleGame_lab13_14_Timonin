# UnscrambleGame

## Описание
Игра, в которой нужно угадать перемешанное слово. За правильный ответ — 20 очков, всего 10 слов. Можно пропускать слово. После прохождения — диалог с итогами и кнопкой «Сыграть снова».

## Технологии
- Kotlin, Jetpack Compose
- ViewModel, StateFlow
- Coroutines, JUnit (тесты)

## Функциональность
- Перемешивание слов
- Проверка ответа без учёта регистра
- Сохранение состояния при повороте экрана (ViewModel)
- Отображение счёта и номера вопроса
- Сообщение об ошибке
- Кнопки «Проверить» и «Пропустить»
- Диалог завершения игры
- Модульные тесты

## Контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?
**Кратко:**  
ViewModel — компонент архитектуры Android, хранящий данные UI и переживающий изменение конфигурации (поворот экрана).  
- **Какую проблему решает?** Без ViewModel данные в Activity/Composable теряются при повороте, так как компонент пересоздаётся. ViewModel сохраняется между этими циклами.  
- **Что происходит без ViewModel?** Счёт, прогресс и другие данные сбрасываются к начальным значениям.

### 2. В чём разница между MutableStateFlow и StateFlow?
- **MutableStateFlow** — изменяемый поток (приватный для ViewModel).  
- **StateFlow** — неизменяемый поток (публичный для UI).  
**Зачем два поля?** Это защищает состояние от случайных модификаций и делает поток данных однонаправленным.

### 3. Почему в ViewModel нельзя хранить Context или View?
Хранение Context/View приводит к утечке памяти, так как ViewModel живёт дольше Activity.  

### 4. Чем StateFlow отличается от remember и rememberSaveable?
- **remember** — временное состояние на время жизни Composable; теряется при повороте.  
- **rememberSaveable** — переживает поворот экрана через Bundle, но не переживает убийство процесса.  
- **StateFlow + ViewModel** — реактивное состояние, сохраняется при повороте, подходит для сложной логики и тестирования.

### 5. Что такое data class в Kotlin и зачем он используется для UI State?
**data class** автоматически генерирует `equals()`, `hashCode()`, `toString()`, `copy()`, `componentN()`.  
**Зачем для UI State:** позволяет создавать неизменяемые объекты, обновлять только нужные поля через `copy()` и упрощает тестирование.

### 6. Как работает collectAsState() в Compose?
`collectAsState()` подписывается на `StateFlow` и возвращает `State`.  
При изменении значения в `StateFlow` Compose автоматически перерисовывает (рекомпозирует) UI, который использует этот `State`.

### 7. Зачем разделять код на пакеты data и ui?
- **data** — модели данных (например, `GameUiState`).  
- **ui** — UI-компоненты и ViewModel.  
Это поддерживает принцип разделения ответственности, упрощает тестирование и масштабирование.

### 8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?
ViewModel уничтожается вместе с процессом, данные не сохраняются.
